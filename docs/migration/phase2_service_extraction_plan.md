# Phase 2 Service Extraction Plan

Static analysis only. This document defines the service-layer extraction plan needed to move business logic out of route handlers while preserving current behavior exactly.

Reference documents:
- [phase0_repository_inventory.md](C:/Users/harikrishnam/Desktop/cms_backend-codex/docs/migration_plan/phase0_repository_inventory.md)
- [phase1_contract_map.md](C:/Users/harikrishnam/Desktop/cms_backend-codex/docs/migration/phase1_contract_map.md)

Extraction rules for this phase:
- No route path, method, redirect target, template name, message string, filename convention, or side effect should change during extraction.
- Route handlers should become orchestration shells only: parse request, call service, translate result to redirect/SSR/JSON/file response.
- Filesystem writes, DB mutation sequences, lock semantics, and editor/processing integration boundaries must remain behavior-compatible.
- Duplicate routes must be documented and wrapped without silently changing which behavior currently wins at runtime.

## 1. Route Logic Extraction Map

In this section, "overloaded" means the route does more than request parsing and response rendering. Pure template renderers such as `GET /login` are intentionally omitted unless they also aggregate state or enforce meaningful access control.

## 3. Repository Extraction Map

Repository extraction should follow service boundaries, not file-by-file rewrites. The goal is to remove direct `db.query(...)` usage from routers first, then from services that still mix persistence with orchestration.

| Service area | Queries currently embedded in routers | Repository candidates | Model ownership boundaries | Transaction-sensitive operations | Risks if repository extraction is delayed |
|---|---|---|---|---|---|
| Auth / session | `User` by username; `User` by username/email OR; `Role` by name/id; `User.count()`; cookie-user lookup by JWT `sub` | `user_repository`, `role_repository` | `user_repository` owns `User`; `role_repository` owns `Role` + role membership count lookups | registration creates roles conditionally, then user, then user-role link; password changes update `password_hash` | auth logic stays scattered across routes and `auth.py`, making browser/API parity brittle |
| Admin users / roles | `User.all()`, `Role.all()`, `UserRole.count()` by role, `User` by id, `Role` by id | `user_repository`, `role_repository` | user lifecycle in `user_repository`; role lookup/counts in `role_repository` | last-admin protection requires consistent count and role replacement in one transaction | duplicate route groups will continue to reload lists and count admins in different ways |
| Projects | `Project.count()`, `Project` list, `Project` by id, `Project` delete, `Project` status update | `project_repository` | `project_repository` owns `Project` CRUD and aggregate counts | project delete may coordinate dependent chapter/file deletions and status updates | routers and current `project_service` continue to diverge on delete behavior |
| Chapters | `Project.chapters`, `Chapter` by id, derived chapter folder flags from `ch.files`, chapter create/delete/rename | `chapter_repository` | `chapter_repository` owns `Chapter`; related file counts come from `file_repository` or a query object | create/rename/delete must align DB and folder operations | directory operations remain tied to ORM mutations in routes |
| Files | `File` by id, `File` by chapter/category/filename, recent `File` list, chapter file list, processed derivative lookup by filename | `file_repository` | `file_repository` owns `File`; lock fields belong with file persistence | file delete, upload overwrite, checkout, processed derivative lookup all touch same model | lock rules, upload rules, and feed queries stay duplicated |
| File versions | recent `FileVersion` list, `FileVersion` create on upload overwrite and processing backup, manual raw SQL delete in `project_service` | `file_version_repository` | `file_version_repository` owns `FileVersion` writes/queries | version rows must be created together with archive copy and file/version mutation | version behavior remains split between upload route, processing route, and `project_service.delete_project_v2` |
| Notifications / activity feeds | recent files ordered by `uploaded_at`; recent versions ordered by `uploaded_at`; per-item project/chapter lookups | `notification_repository`, `activity_repository` or feed query methods on existing repositories | feed repositories own read models only; underlying row ownership remains with file/project/chapter repositories | read-only, but query composition matters for ordering and time-window semantics | N+1 queries remain hidden inside route rendering and are harder to test in isolation |
| Processing orchestration | `File` by id; `Project` and `Chapter` by file foreign keys; processed-file lookup by derived filename; generated-file registration; unlock writes | `processing_repository` is not necessary; use `file_repository`, `file_version_repository`, `project_repository`, `chapter_repository` | processing owns orchestration only; persistence stays with file/version/project/chapter repos | start-process transaction spans lock update, backup copy, version row write, enqueue; background worker spans output registration and unlock | route and background helper remain tightly coupled to ORM session lifecycle |
| Structuring review | `File` by id only, then path derivation | `file_repository` | `structuring_review_service` should not own ORM beyond file lookup | save/export rely on processed path resolution; no complex DB transaction | route keeps hidden storage conventions that frontend and processing both depend on |
| WOPI / editor | `File` by id for every callback and shell render | `file_repository` | `wopi_service` owns protocol, not models | PutFile writes must keep byte persistence and target-path selection consistent | path rules remain duplicated between `wopi.py` and `structuring.py` |

## 4. Side Effect Map

High-risk route groups are listed here by side-effect profile so extraction can preserve hidden behavior, not just explicit outputs.

| Route group | DB side effects | Filesystem side effects | Background execution side effects | External integration side effects | User-visible side effects | Hidden side effects that must be preserved |
|---|---|---|---|---|---|---|
| Browser login / register | login reads user; register may create `Role`, `User`, `UserRole` rows | none | none | JWT creation during login | redirects to `/dashboard` or `/login`; inline template errors | cookie value format is `Bearer <jwt>`; first registered user becomes `Admin`; role bootstrap happens lazily |
| Admin create / role / status / password / delete | creates and mutates `User`, `Role`, `UserRole`; may delete users | none | none | bcrypt for password changes | redirects with `msg` query strings or re-renders with `error` | last-admin protection; self-delete prevention; self-lockout prevention; duplicate route behavior ambiguity |
| Project bootstrap (`POST /projects/create_with_files`) | creates project, chapters, initial file rows | creates project folder tree; writes uploaded files into inferred chapter/category folders | none | none | redirects to `/dashboard` | `team_id=1` hardcoded; `client_name` is patched after base project create; filename regex decides chapter assignment; default chapter fallback is `01` |
| Chapter create / rename / delete / download | creates/updates/deletes `Chapter` | creates category folders, renames chapter folder, deletes chapter tree, creates temp zip | none | none | success redirects and zip downloads | folder names depend on `chapter.number`; duplicate delete route changes message text slightly |
| Chapter upload / versioning | creates `File` and `FileVersion`, updates `File.version`, `uploaded_at`, lock fields | creates category/Archive folders; copies archived version; overwrites or creates files | none | none | redirects back to `?tab={category}&msg=Files+Uploaded+Successfully` | overwrite of locked file by another user is silently skipped, not surfaced; overwrite auto-cancels checkout; archive naming uses `filename_without_ext_v{old_version}.{ext}` |
| File / project delete | deletes `File` or `Project` rows; API delete also cascades chapter/file deletion via service | removes individual file or entire project directory | none | none | redirect or JSON success message | SSR project delete removes disk tree, API delete does not; file delete swallows disk-delete exceptions with `print` |
| Checkout / cancel checkout | updates `File.is_checked_out`, `checked_out_by_id`, `checked_out_at` | none | none | none | redirect messages indicate checked out, locked by other, or cancelled | same-user checkout is effectively idempotent; cancel checkout silently does nothing if actor does not own lock |
| Processing start / background completion | writes `FileVersion`, increments source `File.version`, registers generated `File` rows, unlocks source file | creates archive backup; generated outputs appear beside source; may inject styles into new docx outputs | schedules FastAPI `BackgroundTasks`; background helper unlocks even on failure | processing engines; optional AI structuring inside `StructuringEngine` | immediate JSON `{status:"processing"}`; later chapter page or structuring review reveals outputs | no durable job table in CMS; failure is not exposed via polling; output files always get `version=1`; source `uploaded_at` is not updated |
| Technical scan / apply | scan is read-only; apply registers derived `File` row | reads source docx; writes `_TechEdited` docx | none | `TechnicalEditor` legacy library | technical page loads issue list, then redirects back to chapter page after apply | apply contract only returns `new_file_id`; result file name convention is `_TechEdited` while batch technical process uses `_TechnicallyEdited` |
| Structuring review / save / export | review/export are read-only; save does not write DB, only file bytes | resolves and edits `_Processed.docx` in place; export reads processed file | none | structuring lib helpers; Collabora launch URL depends on WOPI routes | review shell, error page, export download | route computes style catalog but template currently mostly relies on Collabora iframe; explicit save endpoint may be dormant |
| WOPI original edit lifecycle | no DB mutation currently; all writes go to file bytes only | reads and writes original or processed docx directly on disk | none | Collabora invokes CheckFileInfo, GetFile, PutFile | in-browser editor auto-saves; editor shell route provides iframe | WOPI endpoints are not application-authenticated; `SupportsLocks=False`; `UserId` and `UserFriendlyName` are hardcoded |
| Notifications / activities | read-only feed queries | none | none | none | navbar notification dropdown and activities page populate recent events | notification feed is upload-only; activities page performs many per-item lookups and formats relative time in route code |

## 5. Extraction Sequence

The sequence below is intentionally conservative. It prioritizes services that reduce route complexity without changing storage or processing behavior first, then moves into stateful file and processing flows after their dependencies are isolated.

| Stage | Service name | Why it should be extracted at that stage | Prerequisites | Routes affected | Regression risk | Tests required first |
|---|---|---|---|---|---|---|
| 1 | `session_service` | repeated cookie-user gating and viewer DTO assembly are low-risk to centralize and unlock every SSR route | none; relies on existing `get_current_user_from_cookie` | most GET routes in `web.py`, `structuring.py`, `wopi.py` | Low | login/logout smoke tests; SSR redirect tests for anonymous vs authenticated access |
| 2 | `notification_service` | read-only and currently isolated to one JSON endpoint | stage 1 viewer resolution | `GET /api/notifications` | Low | notifications endpoint payload test, anonymous empty-response test |
| 3 | `activity_service` | read-only aggregation removes complex feed building from `web.py` without affecting writes | stage 1 viewer resolution | `GET /activities` | Low-medium | activities page state test with uploads and versions present |
| 4 | `dashboard_service` | dashboard stats are inline placeholders and project DTO assembly is easy to encapsulate | stage 1 viewer resolution; basic project repository methods | `GET /dashboard` | Low-medium | dashboard render test, project list rendering test |
| 5 | `admin_user_service` read-side first, then mutation methods | admin logic is dense but self-contained and independent of file processing | stages 1 and auth/password helpers available | admin list/create/role/status/edit/password/delete routes | Medium | admin access-control tests; last-admin guard test; self-delete/self-lockout tests |
| 6 | `auth_service` | login/register/password logic must be centralized before broader frontend session work, but extraction should follow basic session wrapping | stage 1; user and role repositories in place | `POST /login`, `POST /register`, admin password/create-user flows, `/api/v1/users/login` | Medium | login invalid/valid tests; first-user admin bootstrap test; duplicate registration test |
| 7 | `project_service` read-side methods | project listing/query logic is shared by dashboard, projects page, and API list | dashboard/service repository groundwork | `/dashboard`, `/projects`, `/api/v1/projects/` GET | Medium | project list API test; dashboard/project page state tests |
| 8 | `chapter_service` read-side and simple mutations | chapter list/detail and create/rename/delete are smaller than full upload/version flows, but establish storage-bound chapter ownership | project service read-side; chapter/file repositories | project chapter routes and chapter detail route | Medium-high | chapter create/rename/delete tests; zip download test; chapter detail render test |
| 9 | `file_service` | centralizing file lookup, path resolution, downloads, and deletes is required before versioning and processing extraction | chapter/project repositories; storage path helper conventions documented | file download/delete routes, technical editor shell, thin `/api/v1/files/` | High | file download/delete tests; path resolution tests; thin API upload compatibility test |
| 10 | `version_service` | upload overwrite and processing backup share archive/version behavior; extract once storage access is centralized | file service in place | chapter upload route, processing start/background, technical apply, structuring review/export lookups | High | overwrite upload/version test; processing backup test; derivative registration test |
| 11 | `checkout_service` | lock semantics are reused by upload and processing; extract after file/version storage is stable | file service in place | checkout/cancel routes, processing start, upload overwrite behavior | High | checkout same-user/other-user tests; cancel checkout ownership test |
| 12 | `technical_editor_service` | technical flows are already shell + JSON and can be isolated before the heavier generic processing orchestration | file/version services; permission helper available | technical editor page, scan/apply routes | Medium-high | technical scan contract test; technical apply derivative-file test |
| 13 | `processing_service` | this is the main high-risk orchestration layer and depends on file, version, and lock services being stable first | stages 9-11 complete; engine wrappers unchanged | all processing routes and background helper | Very high | start-process lock/backup test; generated-file registration test; failure-unlocks-file test; structuring polling test |
| 14 | `structuring_review_service` | processed-file resolution and review/export become easier once processing naming and versioning are centralized | processing + version services stable | all structuring review/save/export routes | High | processed-file resolution test; export test; explicit save test |
| 15 | `wopi_service` | WOPI callbacks are the highest-risk integration boundary and should move only after path rules and processed/original target semantics are stable | structuring review and file path helpers stable | all routes in `app/routers/wopi.py` | Very high | CheckFileInfo contract test; GetFile/PutFile roundtrip test; structuring WOPI target-path tests |

## 6. Deferred Areas

These areas should not be extracted early, even if they contain a lot of logic, because they depend on multiple unstable boundaries at once.

| Deferred area | Services / routes | Why it is high risk | What must stabilize first |
|---|---|---|---|
| Processing orchestration | `processing_service`; all routes in `app/routers/processing.py` | combines permission policy, lock semantics, versioning, background execution, multiple engines, and output registration | file storage paths, version creation, checkout semantics, repository ownership of `File` and `FileVersion` |
| WOPI original and structuring callbacks | `wopi_service`; all routes in `app/routers/wopi.py` | writes bytes directly to disk, must remain compatible with Collabora expectations, and shares target-path rules with structuring review | path-resolution helpers, processed/original naming rules, review service contract |
| Structuring review shell and save/export | `structuring_review_service`; routes in `app/routers/structuring.py` | depends on processed output naming from processing, WOPI callback reachability, Collabora base URL config, and potentially dormant explicit save behavior | processing output registration, WOPI target-path policy, route ownership of review/export |
| Project bootstrap with initial files | `project_service.create_project_with_initial_files`; `POST /projects/create_with_files` | spans project, chapter, file, and storage path creation with filename inference and fallback chapter assignment | project repository, chapter repository, file service, storage path helper |
| Upload overwrite/versioning | `file_service` + `version_service`; `POST /projects/{project_id}/chapter/{chapter_id}/upload` | tightly couples archive writes, version rows, lock release, and partial-success behavior | basic file service, version repository, checkout service semantics |
| Delete lifecycle normalization | `project_service.delete_project`, `file_service.delete_file`, `chapter_service.delete_chapter` | current SSR/API delete behaviors are intentionally inconsistent; premature extraction can accidentally normalize behavior | explicit delete contract inventory, repository ownership, filesystem delete helpers |
| Duplicate admin route groups | password/delete duplicates in `web.py` | changing handler ownership too early can silently change effective validation or redirect behavior | route shadowing must be verified in tests first; admin user service should wrap current runtime behavior before deduplication |

## 7. Deliverable Closure

### Top 10 safest service extractions first

1. `session_service`
2. `notification_service`
3. `activity_service`
4. `dashboard_service`
5. `auth_service` cookie/token helpers only
6. `admin_user_service` read-side methods
7. `project_service` read-side methods
8. `chapter_service` read-side methods
9. `technical_editor_service` page-state builder
10. `file_service` download lookup helpers

### Top 10 highest-risk route handlers

1. `POST /projects/create_with_files`
2. `POST /projects/{project_id}/chapter/{chapter_id}/upload`
3. `POST /api/v1/processing/files/{file_id}/process/{process_type}`
4. `POST /projects/{project_id}/delete`
5. `DELETE /api/v1/projects/{project_id}`
6. `POST /projects/files/{file_id}/checkout`
7. `POST /projects/files/{file_id}/cancel_checkout`
8. `GET /api/v1/files/{file_id}/structuring/review`
9. `POST /wopi/files/{file_id}/contents`
10. `POST /wopi/files/{file_id}/structuring/contents`

### Top 10 queries that should move to repositories

1. `db.query(models.User).filter(models.User.username == username).first()` in login and API auth flows
2. `db.query(models.User).filter((models.User.username == username) | (models.User.email == email)).first()` in SSR/admin registration flows
3. `db.query(models.Role).filter(models.Role.name == "Viewer").first()` and similar role-bootstrap lookups
4. `db.query(models.Project).offset(skip).limit(limit).all()` used for dashboard/projects/API listing
5. `db.query(models.Project).filter(models.Project.id == project_id).first()` repeated across project/chapter/file flows
6. `db.query(models.Chapter).filter(models.Chapter.id == chapter_id).first()` repeated across chapter lifecycle flows
7. `db.query(models.File).filter(models.File.chapter_id == chapter_id, models.File.category == category, models.File.filename == upload.filename).first()` in upload overwrite detection
8. `db.query(models.File).filter(models.File.id == file_id).first()` repeated across download/delete/checkout/processing/technical/review/WOPI flows
9. `db.query(models.FileVersion).order_by(models.FileVersion.uploaded_at.desc()).limit(50).all()` in activities page aggregation
10. `db.query(models.UserRole).filter(models.UserRole.role_id == admin_role.id).count()` for last-admin protection

### Top 10 side effects that must be preserved exactly

1. Browser login must set `access_token` cookie as `Bearer <jwt>`.
2. The first registered user must still receive the `Admin` role.
3. Project bootstrap must still create chapters `01..N` and default unmatched files to chapter `01`.
4. Upload overwrite must still archive the old file into `Archive/filename_v{old_version}.{ext}` before overwriting.
5. Upload overwrite must still auto-release the file lock after successful replacement.
6. Checkout by another user must still redirect back with `msg=File+Locked+By+Other`.
7. Processing start must still lock the file, create a backup, increment `File.version`, and return `{status:"processing"}` immediately.
8. Processing background failure must still unlock the source file even when work fails.
9. Structuring review and WOPI structuring mode must still target `_Processed.docx`, not the original file.
10. SSR project delete must still remove the project directory tree before redirecting to `/dashboard?msg=Book+Deleted`.

### Recommended extraction order

1. Extract `session_service` and replace repeated viewer-context assembly in SSR routes.
2. Extract `notification_service` and `activity_service` because they are read-only and isolated.
3. Extract `dashboard_service` so project list pages stop assembling view models in `web.py`.
4. Extract `admin_user_service` read paths, then mutation paths, while preserving duplicate-route behavior.
5. Extract `auth_service` so login, register, admin create-user, and password-change logic stop living in routes.
6. Extract `project_service` read-side methods, then its create/delete orchestration methods.
7. Extract `chapter_service` for chapter listing, create, rename, delete, and zip packaging.
8. Extract `file_service` for file lookup, download, delete, and storage path helpers.
9. Extract `version_service` so archive/version behavior becomes shared between upload, processing, and derivative-file flows.
10. Extract `checkout_service` so lock semantics stop being duplicated in `web.py` and `processing.py`.
11. Extract `technical_editor_service` for scan/apply orchestration and page-state construction.
12. Extract `processing_service` last among core CMS services, keeping engines and FastAPI `BackgroundTasks` behavior unchanged.
13. Extract `structuring_review_service` once processed-file naming and versioning are stable.
14. Extract `wopi_service` last, preserving exact CheckFileInfo/GetFile/PutFile behavior and target-path semantics.
