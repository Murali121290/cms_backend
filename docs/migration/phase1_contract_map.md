# Phase 1 Contract Map

Static analysis only. This document defines compatibility-first backend contracts, page-state contracts, and service boundaries for the current FastAPI CMS so migration can proceed without breaking existing behavior.

Reference inventory: [phase0_repository_inventory.md](C:/Users/harikrishnam/Desktop/cms_backend-codex/docs/migration_plan/phase0_repository_inventory.md)

Contract type legend:
- `SSR-only`: route remains server-rendered and is not expected to become a typed frontend contract in Phase 1.
- `SSR shell + JSON endpoints`: route remains as a shell/layout/entry page, while state and mutations move behind explicit JSON contracts.
- `JSON API for typed frontend`: route becomes or remains an explicit, typed data/mutation contract for frontend consumers.
- `integration endpoint`: route is owned by backend/integration boundaries and should not be frontend-driven beyond navigation/wrapping.

Disposition legend:
- `remain as-is`: keep active with minimal contract wrapping.
- `split`: separate UI concerns from domain/data concerns while preserving route compatibility.
- `wrap`: keep route but define a stable service/page-state contract around it.
- `deprecate later`: keep temporarily for compatibility, but plan to replace with a normalized contract after parity is proven.

## 2. Request / Response Schema Map

Normalization format:
- `Current request inputs`: what the code actually accepts today.
- `Current response shape`: what the route actually returns today.
- `Proposed normalized schema`: additive stable contract that can preserve current behavior while making typed consumption possible.
- Error schemas are written as compatibility-first JSON contracts even when the current route sometimes redirects or renders HTML.

### Auth and identity endpoints

| Route | Current request inputs | Current response shape | Missing schema risks | Hidden implicit fields / assumptions | Proposed normalized request schema | Proposed normalized response schema | Error response schema | Status / polling schema | Compatibility notes |
|---|---|---|---|---|---|---|---|---|---|
| `POST /login` | form `username`, `password` | 302 redirect to `/dashboard` + `access_token` cookie, or `login.html` with `error` | no explicit form schema object, no session payload | cookie stores `Bearer <jwt>` rather than raw token; no `next`/redirect input | `{ username: str, password: str, redirect_to?: str }` | Browser-mode contract: `302` + cookie; optional parallel JSON contract `{ status:"ok", session:{ username, roles[], expires_at? }, redirect_to:"/dashboard" }` | `{ status:"error", code:"INVALID_CREDENTIALS" | "AUTH_FAILURE", message:str, field_errors?:{ username?:str, password?:str } }` | n/a | Preserve current POST+redirect flow; normalized JSON can be additive behind content negotiation or parallel endpoint later. |

## 6. Normalization Priorities

Contracts that must be normalized before any frontend rewrite:

| Priority item | Why normalization is required | Current inconsistency | Target stable contract | Migration risk if left unresolved |
|---|---|---|---|---|
| Auth / session | Every page and API flow depends on viewer identity; frontend migration needs a stable bootstrap shape | Cookie auth is implicit for SSR, bearer auth is explicit for APIs, and there is no cookie JSON session endpoint | One documented auth/session contract with browser session semantics, bearer semantics, and a stable `viewer/session` DTO | High: a new frontend can easily authenticate for APIs but still fail on SSR-linked flows or redirects |
| Delete behavior | Delete actions affect user trust and storage integrity | SSR project delete removes filesystem + DB; API delete removes DB only; chapter/file delete capture redirect context differently | Canonical delete lifecycle per resource with explicit disk cleanup rules and return payloads | High: partial deletion or orphaned disk data during cutover |
| File upload / versioning | Upload is central to CMS behavior and carries hidden archive semantics | SSR upload handles overwrite/archive/version/unlock; `/api/v1/files/` only saves a file record with a different path strategy | Canonical upload/version contract returning created/replaced/skipped items and archive metadata | High: data loss, broken version history, or mismatched file locations |
| Processing status | Frontend orchestration cannot depend on implicit filename discovery alone | Processing start has no job ID and status polling only returns `processing|completed` for structuring based on `_Processed` file presence | Stable status contract with source file, process type, status, and optional derived file metadata | High: impossible to build reliable typed progress UI or retry/error handling |
| Admin route duplication | Duplicate handlers make route behavior order-dependent and page contracts ambiguous | `/admin/users/{user_id}/password` and `/admin/users/{user_id}/delete` are each defined twice with slightly different behavior/context | Single documented contract per admin mutation path, with one page-state DTO for password/change/delete flows | High: frontend parity can be built against the wrong effective behavior |
| Root route ownership | Entry-point routing determines how browser sessions land in the app | `web.home` redirects by cookie state while `main.read_root` returns JSON on the same `/` path | One explicit owner for `/` plus a separate health/API root if needed | High: new frontend shell or proxy cutover can accidentally replace login/dashboard routing |
| WOPI boundaries | Editor flows must remain backend-owned and protocol-safe | Browser routes are authenticated, but WOPI endpoints are unauthenticated integration endpoints and write directly to disk | A documented integration boundary separating browser shells from WOPI callback endpoints and their file targets | High: editor breakage, accidental auth enforcement on Collabora callbacks, or insecure exposure |
| Project create split | Project creation is currently two different products: thin API and full SSR workflow | `/api/v1/projects/` creates only a project row; `/projects/create_with_files` creates project, chapters, files, and directories | Explicitly distinguish `project-core create` from `project-bootstrap create-with-files` contracts | Medium-high: frontend could call the wrong endpoint and silently miss required side effects |
| Checkout / lock semantics | Lock state influences upload, processing, and edit behavior | Lock rules live partly in checkout routes and partly in processing start; stale-lock policy is absent | One lock contract covering acquire, release, idempotence, and conflict responses | Medium-high: race conditions or broken optimistic UI during migration |
| Technical editor result schema | The current UI already behaves like an app, but its data shape is implicit | scan response is a dynamic dict with undocumented keys; apply response returns only `new_file_id` | Stable issue list / replacement submission / result DTOs | Medium: page modernization work will otherwise re-encode hidden assumptions from template JS |
| Notification / activity feed shape | Shared nav and activity page will likely move earlier than deep editor flows | current feed payloads are presentation-only and ID-less; activity page re-queries related rows per item | stable feed DTOs with IDs, timestamps, entity links, and item types | Medium: incremental frontend components will need ad hoc transformations and duplicate logic |
| Team API model drift | A typed frontend cannot safely depend on an API that writes nonexistent ORM fields | `team_service.create_team` references `description` and `owner_id` not present on `Team` model | Explicit compatibility note that team API is non-canonical until its contract matches persistence | Medium: any new UI using teams will fail unpredictably |

## 7. Deliverable Closure

### Top 15 routes requiring contract normalization

1. `GET /` because `web.home` and `main.read_root` compete for root ownership with different response modes.
2. `POST /login` because it defines the browser session contract and cookie format for all SSR flows.
3. `POST /projects/create_with_files` because it is the only route that creates project, chapters, directories, and initial file placements together.
4. `POST /projects/{project_id}/chapters/create` because it combines DB writes with mandatory directory creation.
5. `POST /projects/{project_id}/chapter/{chapter_id}/rename` because chapter numbering changes also rename storage paths.
6. `POST /projects/{project_id}/chapter/{chapter_id}/delete` because it is duplicated and includes filesystem deletion.
7. `POST /projects/{project_id}/chapter/{chapter_id}/upload` because it owns overwrite detection, archive creation, version rows, and auto check-in.
8. `POST /projects/files/{file_id}/checkout` because downstream editing and processing assume its lock semantics.
9. `POST /projects/files/{file_id}/cancel_checkout` because it is the only explicit unlock path exposed to users.
10. `POST /projects/files/{file_id}/delete` because it deletes disk state and redirects based on captured category context.
11. `POST /projects/{project_id}/delete` because it removes the project directory and DB row.
12. `DELETE /api/v1/projects/{project_id}` because it is not side-effect-equivalent to the SSR delete route.
13. `POST /api/v1/processing/files/{file_id}/process/{process_type}` because it locks, versions, dispatches work, and registers outputs without a job ID.
14. `GET /api/v1/processing/files/{file_id}/structuring_status` because typed polling depends on this minimal implicit status contract.
15. `POST /wopi/files/{file_id}/contents` because it is the write path for in-browser editing and bypasses user-facing route auth.
